## Kafka Consumer Lag Monitoring

### Topic Monitoring

Monitor consumer lag for the following Kafka topics:

```yaml
Topics to Monitor:
  - tmdb.movies.raw
  - tmdb.tv_series.raw
  - tmdb.people.raw
```

### Checking Consumer Lag

<Tabs>
  <Tab title="Using Kafka CLI">
    ```bash
    # Check consumer group lag
    kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
      --group stream-processor-group \
      --describe
    
    # Monitor specific topic
    kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
      --group stream-processor-group \
      --topic tmdb.movies.raw \
      --describe
    ```
  </Tab>
  <Tab title="Using Monitoring Tools">
    Configure your Kafka monitoring tool (e.g., Kafka Manager, Confluent Control Center) to track:
    - Current offset
    - Log end offset
    - Lag (difference between log end and current offset)
    - Consumer group state
  </Tab>
</Tabs>

<Warning>
  **High Consumer Lag Alert**: If consumer lag exceeds 10,000 messages, investigate streaming job performance and resource allocation.
</Warning>

### Critical Kafka Metrics

| Metric | Description | Alert Threshold |
|--------|-------------|----------------|
| Consumer Lag | Messages pending processing | > 10,000 |
| Offset Commit Rate | How often offsets are committed | < 1/min |
| Fetch Rate | Messages consumed per second | Drops > 50% |
| Error Rate | Failed message processing | > 1% |

## Spark Job Monitoring

### Stream Processing Jobs

The platform runs continuous Spark Structured Streaming jobs that process Kafka events:

**Job Name**: `KafkaStreamToDelta` (stream_processor/main.py:41)

<Accordion title="Key Metrics to Track">
  - **Input Rate**: Records/second from Kafka sources
  - **Processing Rate**: Records/second being processed
  - **Batch Duration**: Time to process each micro-batch
  - **Total Delay**: Scheduling delay + processing time
  - **Active Queries**: Number of running streaming queries (expected: 6 - valid/invalid for each entity type)
</Accordion>

### Batch Processing Jobs

Monitor these Spark batch jobs:

1. **Deduplication Job** (`run_dedup_timestamp`)
   - Removes duplicate records based on timestamp
   - Runs on Bronze to Silver layer transformation
   - Location: batch_jobs/pipelines/bronze_silver/minio_to_minio.py:19

2. **ClickHouse Export Job** (`write_to_clickhouse`)
   - Exports data from Delta Lake to ClickHouse
   - Location: batch_jobs/pipelines/silver_silver/minio_to_clickhouse.py:80

3. **Neo4j Graph Job** (`write_to_neo4j`)
   - Builds knowledge graph in Neo4j

4. **Pinecone Vector Job** (`write_to_pinecone`)
   - Indexes embeddings in Pinecone

### Accessing Spark UI

<Steps>
  <Step title="Access Local Spark UI">
    When running locally, access the Spark UI at:
    ```
    http://localhost:4040
    ```
  </Step>
  <Step title="Monitor Streaming Queries">
    Navigate to the **Structured Streaming** tab to view:

  </Step>

</Tip>

**Key Paths to Monitor**:

- Silver Layer Valid: `s3a://deltalake/silver/valid/{movies,tvseries,people}`

### ClickHouse

**Connection**: localhost:8123 (HTTP), localhost:9000 (Native)

```sql
-- Check system health
SELECT * FROM system.metrics;

-- Monitor query performance
SELECT query, event_time, query_duration_ms 
FROM system.query_log 
WHERE query_duration_ms > 1000 
ORDER BY event_time DESC 
LIMIT 10;

-- Check table sizes
SELECT 
    table,
    formatReadableSize(sum(bytes)) as size,
    sum(rows) as rows
FROM system.parts
WHERE active AND database = 'silver_layer'
GROUP BY table;

-- Verify data freshness
SELECT 
    max(updated_at) as last_update,
    count(*) as total_records
FROM silver_layer.movies;
```

### Redis (Batch Version Tracking)

## Log Locations and Formats

### Logging Configuration

All services use centralized logging configuration (common/logging_config.py:5):

```python
Format: "%(asctime)s | %(levelname)s | %(name)s | %(message)s"
Output: stdout
Default Level: INFO (configurable via LOG_LEVEL env var)
```

### Log Levels

Set the `LOG_LEVEL` environment variable:

<Tabs>
  <Tab title="Development">
    ```bash
    export LOG_LEVEL=DEBUG
    ```
    Use for detailed troubleshooting and development
  </Tab>
  <Tab title="Production">
    ```bash
    export LOG_LEVEL=INFO
    ```
    Standard production logging
  </Tab>
  <Tab title="Critical Only">
    ```bash
    export LOG_LEVEL=ERROR
    ```
    Minimal logging for high-performance scenarios
  </Tab>
</Tabs>

### Component-Specific Logs

<AccordionGroup>
  <Accordion title="Stream Processor Logs">
    **Component**: Stream processor (Kafka to Delta Lake)
    
    **Location**: stdout/stderr or configured log aggregation
    
    **Key Log Messages**:
    - `"Loading configuration..."` - Service startup
    - `"Spark session started"` - Spark initialization
    - `"Creating stream logic for topic: {topic}"` - Stream setup
    - `"Batch {batch_id} count={count}"` - Micro-batch processing
    
    **Source**: stream_processor/main.py:18, stream_processor/sinks/delta_lake_sink.py:17
  </Accordion>
  
  <Accordion title="Batch Job Logs">
    **Component**: Batch processing jobs
    
    **Key Log Messages**:
    - `"Loading configuration..."` - Job initialization
    - `"Last version for {key} found in Redis: {version}"` - Batch version tracking
    - `"Reading data from {path} with version {start} to {end}"` - Data loading
    - `"Upserting data from {source} to {target}"` - Data transformation
    - `"CRITICAL: Last version for {key} NOT found in Redis"` - Critical error
    
    **Source**: batch_jobs/pipelines/bronze_silver/minio_to_minio.py:17-69
  </Accordion>
  
  <Accordion title="Ingestion Service Logs">
    **Component**: Data ingestion from TMDB API
    
    **Key Log Messages**:
    - `"Loading config with env={env}"` - Configuration loading
    - `"Config file not found: {path}"` - Configuration errors
    - Producer retry and buffer warnings
  </Accordion>
  
  <Accordion title="ClickHouse Writer Logs">
    **Component**: ClickHouse data export
    
    **Key Log Messages**:
    - `"Start writing to Clickhouse table: {database}.{table}"` - Write start
    - `"Finish writing to Clickhouse table: {database}.{table}"` - Write success
    - `"Error when write to Clickhouse table: {database}.{table}"` - Write failure
    
    **Source**: batch_jobs/io/writers/clickhouse_jdbc_writer.py:29-43
  </Accordion>
</AccordionGroup>

### Log Aggregation

<Tip>
  For production deployments, consider aggregating logs using:
  - **ELK Stack** (Elasticsearch, Logstash, Kibana)
  - **Loki** with Grafana
  - **CloudWatch** (for AWS deployments)
  - **Datadog** or **Splunk** for enterprise monitoring
</Tip>

### Sample Log Analysis Queries

```bash
# Find all errors in last hour
grep "ERROR" app.log | grep "$(date -d '1 hour ago' '+%Y-%m-%d %H'):"

# Count processing batches
grep "Batch.*count=" stream_processor.log | wc -l

# Find configuration loading issues
grep "Config file not found" *.log

# Monitor Redis version tracking
grep "Last version.*found in Redis" batch_jobs.log
```
