This reference documents all schema definitions used across the platform, from raw event schemas to entity schemas for movies, TV series, and people.

## Event Schemas

Event schemas define the structure of incoming data from Kafka streams.

### Partial Event Schema

Base event metadata for all incoming events:

<ParamField path="data_type" type="String" required>
  Type of entity ("movie", "tv_series", or "person")
</ParamField>

<ParamField path="data_label" type="String" required>
  Label or classification for the data
</ParamField>

<ParamField path="timestamp" type="Timestamp" required>
  When the event was generated or ingested
</ParamField>

```python
# Source: stream_processor/schema/event_schema.py:3-7
PARTIAL_EVENT_SCHEMA = StructType([
    StructField("data_type", StringType(), True),
    StructField("data_label", StringType(), True),
    StructField("timestamp", TimestampType(), True),
])
```

## Schema Evolution

The platform supports schema evolution with these principles:

<AccordionGroup>
  <Accordion title="Adding Fields">
    New optional fields can be added to any schema without breaking existing pipelines. All fields are nullable by default.
  </Accordion>
  
  <Accordion title="Renaming Fields">
    Field renames require updating all downstream transformations and should be done with a migration strategy.
  </Accordion>
  
  <Accordion title="Removing Fields">
    Deprecated fields should be marked as nullable first, then removed after ensuring no downstream dependencies.
  </Accordion>
  
  <Accordion title="Changing Types">
    Type changes require careful migration:
    - Widening types (Int → Long) is safe
    - Narrowing types requires data validation
    - String ↔ Structured changes need custom migration
  </Accordion>
</AccordionGroup>

## Validation Rules

Schema validation occurs at multiple stages:

### Bronze Layer Validation

- **Required fields**: `data_type`, `timestamp` must be present
- **Data type**: Must be one of: "movie", "tv_series", "person"
- **JSON structure**: Must be valid JSON

### Silver Layer Validation

- **Schema conformance**: Data must match entity schema
- **ID validation**: Primary IDs must be positive integers
- **Array validation**: Nested arrays must contain valid structs
- **Date validation**: Date strings must be valid YYYY-MM-DD format

### Gold Layer Validation

- **Referential integrity**: Foreign keys must reference existing entities
- **Uniqueness**: Primary keys must be unique in target systems
- **Type constraints**: Values must fit target system types (e.g., ClickHouse Int64)
